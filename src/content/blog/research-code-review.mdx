기업에서는 코드 리뷰가 일상이다. 다른 팀원이 내 코드를 읽고, 피드백을 주고, 승인한다. 연구실에서는 이 문화가 흔하지 않다. 대부분 혼자 쓰고 혼자 돌린다.

## 연구 코드의 특성

연구 코드는 기업 코드와 다르다.

**일회성**: 논문 한 편을 위한 실험 코드. 반복 재사용을 전제로 하지 않는다.

**빠른 개발**: 아이디어를 빨리 테스트해야 한다. 꼼꼼한 설계보다 "돌아가면 OK".

**작성자 = 사용자**: 대부분 본인이 쓰고 본인이 돌린다. 다른 사람이 쓸 일이 적다.

이런 이유로 연구 코드는 보통 지저분하다.

## 코드 리뷰의 필요성

그래도 가끔 코드 리뷰가 필요하다.

**공동 연구**: 여러 사람이 같은 코드베이스를 쓸 때. 다른 사람이 내 코드를 이해할 수 있어야 한다.

**재현성**: 논문을 함께 쓰는 공저자가 실험을 재현해보려고 할 때. 코드가 실행 가능해야 한다.

**실수 방지**: 혼자 쓰면 버그를 못 발견한다. 다른 사람이 보면 명백한 실수가 눈에 띈다.

## 처음 리뷰 받을 때

누군가 내 코드를 처음 봤을 때의 기분은 부끄러움이었다. 변수명 엉망, 주석 없음, 중복 코드 범벅.

"이거 너무 지저분해서 미안해"라고 먼저 사과하게 된다. 리뷰어는 "다 그래"라고 말하지만, 속으로는 어떻게 생각할지 모른다.

## 피드백의 종류

**스타일**: "변수명을 더 명확하게", "주석 추가 필요". 쉬운 피드백.

**구조**: "이 함수를 나누는 게 낫다", "클래스로 만드는 건 어때?". 구조적 개선.

**로직**: "여기 버그가 있는 것 같다", "이 조건이 누락됐다". 가장 중요한 피드백.

**성능**: "이 부분을 벡터화하면 빨라진다", "이 루프를 개선할 수 있다". 최적화.

## 피드백 받는 자세

방어적으로 반응하면 안 된다. "내가 생각이 있어서 이렇게 쓴 거야!"라고 말하고 싶지만, 대부분 그냥 안 생각한 경우다.

리뷰어의 의견을 일단 듣고, 맞는 건 받아들이고, 이견이 있으면 차분히 논의하자.

## 리뷰하는 쪽

반대로 다른 사람의 코드를 리뷰해본 적도 있다. 이것도 경험이다.

**이해가 안 되는 건 일단 질문**: "이 부분이 왜 이렇게 되어 있어요?" 답변을 듣고 판단한다.

**건설적으로**: "이건 틀렸다"보다 "이렇게 하면 어떨까요?" 제안형으로.

**사소한 것도 중요**: 변수명 하나, 주석 하나도 가독성에 영향을 준다.

## 결론

연구 코드 리뷰는 연구 문화에 좋은 영향을 준다. 코드 품질이 올라가고, 재현성이 높아지고, 서로 배운다.

하지만 문화로 자리 잡지 않은 연구실이 많다. 누군가 시작해야 하는데, 그게 쉽지 않다. 나도 내 연구실에서 이걸 도입할 수 있을까 고민 중이다.
