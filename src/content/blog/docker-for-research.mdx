연구 코드의 가장 큰 문제 중 하나는 재현성이다. "몇 달 전에 돌아갔는데 지금은 왜 안 되지?" 같은 경험을 여러 번 했다. 이걸 해결하는 방법 중 하나가 Docker다.

## 왜 재현이 안 되는가

**라이브러리 버전**: PyTorch 1.9에서 돌던 코드가 2.0에서 안 돌 수 있다.

**운영체제 차이**: Linux에서 되던 게 macOS에서 안 되거나, 그 반대.

**하드웨어 차이**: GPU 종류에 따라 결과가 미묘하게 달라진다.

**환경 변수**: 설정 파일, 환경 변수, 경로. 이런 게 다르면 다르게 돌아간다.

## Docker가 뭔가

Docker는 애플리케이션을 "컨테이너"라는 격리된 환경에 담는 도구다. 필요한 모든 의존성(라이브러리, 설정, 시스템 도구)을 하나의 이미지로 묶는다.

이 이미지를 다른 컴퓨터에서 실행하면, 같은 환경에서 같은 코드가 돌아간다.

## 연구에서의 활용

**환경 고정**: 논문 작성 시점의 라이브러리 버전을 고정. 1년 후에 리뷰어가 "돌려봐라"라고 해도 같은 결과가 나온다.

**공동 연구**: 다른 연구자와 코드를 공유할 때, Docker 이미지를 주면 설치 과정 없이 바로 실행 가능.

**서버 이전**: 새 서버로 옮겨도 환경 재구축에 시간을 안 쓴다.

## 간단한 예시

`Dockerfile` 하나를 만든다.

```
FROM python:3.10
RUN pip install torch numpy pandas matplotlib
COPY . /app
WORKDIR /app
CMD ["python", "main.py"]
```

이걸 빌드하면 이미지가 된다. 이미지를 저장하거나 공유하면, 다른 컴퓨터에서도 같은 환경이 된다.

## 단점

**학습 곡선**: Docker를 처음 배우는 건 시간이 든다. 명령어, 개념, 문제 해결.

**용량**: 이미지 하나가 수GB. 쌓이면 디스크가 금방 찬다.

**GPU 설정**: GPU를 쓰려면 추가 설정이 필요하다. nvidia-docker 같은 것.

**간단한 경우에는 과하다**: 작은 스크립트 하나 돌리려고 Docker를 쓰는 건 배보다 배꼽이 크다.

## 현실적인 팁

모든 연구 프로젝트에 Docker를 쓸 필요는 없다. 중요한 프로젝트, 재현성이 꼭 필요한 프로젝트에만 쓰자.

대신 모든 프로젝트에 `requirements.txt`나 `environment.yml`은 만들어두자. 이것만으로도 상당 부분의 문제가 해결된다.

## 결론

연구의 재현성은 점점 중요해지고 있다. 저널에서도 코드 공개를 요구하는 곳이 늘어난다. Docker 같은 도구를 알아두면 이런 흐름에 대응하기 쉽다.

한 번 배워두면, 평생 유용한 도구다. 시작은 어려워도.
